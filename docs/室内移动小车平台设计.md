# 室内移动小车平台设计（草案）

面向室内场景的一台**移动底盘 + 机械臂 + 摄像头 + 物品识别与定位**的集成设计，便于后续对接仿真、实机或 agent 工具层。

---

## 1. 系统总览

| 子系统 | 功能 | 与整车的关系 |
|--------|------|----------------|
| **底盘（4 轮）** | 前后移动、左右平移或等效「到达侧向」能力 | 建立车体坐标系、里程/位姿估计 |
| **机械臂** | 抓取、放下（释放） | 末端相对车体有固定几何链；抓取点在车体坐标系中可算 |
| **摄像头** | 拍照（单帧/短序列） | 与车体、腕部外参标定；图像进识别模块 |
| **识别与定位** | 识别物体类别/实例，并给出**可导航的目标表示** | 输出给路径规划 → 底盘执行 |

建议软件上拆成四个**逻辑设备**，每个设备暴露原子能力（便于 agent 一个 tool 一类动作）：

```
摄像头 ──拍照──► 图像
                    ▼
识别与定位 ──► { 物体 id, 类别, 位姿或观测 }
                    ▼
底盘 ◄── 路径/速度指令（到目标附近）
机械臂 ◄── 抓取/放下（对准后执行）
```

---

## 2. 底盘：四轮与「前后左右」

「前后左右」在工程上有两种常见实现，需二选一（或分档：简单版 / 全向版）。

### 方案 A：差速/阿克曼为主（成本低、易做）

- **前后**：左右轮同速正反转。
- **左右**：没有真正横移，用 **原地旋转 + 再直行** 等效到侧向目标（室内足够时常用）。
- 4 轮布局常见：**前舵轮 + 后固定** 或 **左右差速两轮驱动 + 万向轮**（视觉上仍是 4 轮着地）。

适合：走廊、空地，对「严格横移」要求不高。

### 方案 B：全向底盘（麦克纳姆轮 / 全向轮）

- **前后、左右、斜向**：通过四轮速度矢量合成，实现平面内任意方向速度（在摩擦与标定允许范围内）。
- 硬件与控制复杂度更高，需要标定轮系运动学矩阵。

适合：工位间频繁侧移、狭窄区域横向微调。

### 底盘应输出的状态（给上层用）

- **位姿**：\((x, y, \theta)\) 在地图或 SLAM 坐标系下（或起步原点为基准的里程系，并说明漂移）。
- **速度反馈**（可选）：线速度、角速度，用于闭环与安全限速。

### 原子动作建议（与现有 `car_control` 可演进对齐）

| 动作 | 含义（方案 A） | 含义（方案 B） |
|------|----------------|----------------|
| `forward` / `backward` | 沿车头前后 | 沿车头前后 |
| `turn_left` / `turn_right` | 绕垂直轴旋转 | 绕垂直轴旋转 |
| `strafe_left` / `strafe_right` | 可用「转 + 走」等效 | 真正横移（若采用全向轮） |
| `stop` | 全部轮速清零 | 同上 |

若短期只做方案 A，可将「左右」在协议里定义为 **`strafe_*` = 组合宏**（内部拆成转 + 直行），对上层仍保持一个接口。

## 3. 机械臂：抓取与放下

### 能力边界

- **抓取**：移动到预姿态 → 闭合夹爪/吸盘 → 可选「微提拉」确认负载。
- **放下**：移动到放置位姿 → 张开夹爪 → 臂收回安全姿态。

### 建议接口（逻辑层）

- `arm_move_to(pose)`：关节角或笛卡尔目标（在车体或世界系下需约定）。
- `arm_grasp()` / `arm_release()`：在已对准前提下执行末端动作。
- `arm_home()`：回收纳/行驶姿态，避免臂与障碍物干涉。

### 与底盘的协作顺序（典型）

1. 底盘粗定位到物体附近（由识别与定位给出目标）。
2. 停车 `stop`。
3. 臂展开、视觉伺服（可选）微调对准。
4. `arm_grasp` → 可选再 `拍照` 确认。

### 拾放双目标时：收纳筐何时再定位（重要）

若任务为「先抓 A，再放到 B（如收纳筐）」：

- **第一次拍照 + 识别**适合优先锁定 **被抓物体 A**（如黄色衣服），用于「靠近 A → 抓取」。
- 在 **底盘已移动、已抓取 A 之后**，车体坐标系与视野内容已变；**起步时同一张图里给出的 B（收纳筐）位姿，不应再直接用于「驶向筐」的导航**（误差大甚至指向错误方向）。
- 正确做法：在 **抓取完成之后**（或至少在车体已离开原站位之后）**再拍一帧/一段**，并 **单独对收纳筐做一次识别与定位**，用该结果规划「靠近筐 → 释放」。

因此高层步骤上，「收纳筐的可导航精定位」应落在 **抓取之后、驶向筐之前**，而不是与「只服务抓衣」的第一次识别强行绑在同一步里。

## 4. 摄像头：拍照

### 硬件与安装

- 固定在车体前方或腕部末端（**腕部**更利于对准物体，但外参随关节变；**车体固定**标定简单）。
- 同步：拍照时间戳与当前里程计/关节角对齐，便于把检测框变到车体系。

### 原子能力

- `camera_capture()` → 返回 `{ image_id, timestamp, frame_meta }`。
- 可选 `camera_stream_start/stop`（若识别在边端做流式）。

---

## 5. 识别物品 + 返回「定位」（给小车到达用）

目标：**识别物体**后，上层能规划出一条让底盘**靠近可抓取区域**的轨迹。

### 5.1 输出应包含什么（建议统一 JSON）

至少区分两层，避免把「像素」和「米」混用：

```json
{
  "object_id": "cup_001",
  "label": "纸杯",
  "confidence": 0.91,
  "frame": "camera_front",
  "image_uv": { "u": 512, "v": 380 },
  "bearing": { "azimuth_deg": 12.3, "elevation_deg": -5.1 },
  "range_m": 1.25,
  "pose_vehicle": { "x_m": 0.45, "y_m": -0.12, "theta_deg": 0 },
  "pose_map": { "x_m": 3.2, "y_m": 1.1, "theta_deg": 45, "frame_id": "map" },
  "quality": "mono_estimate"
}
```

字段说明：

| 字段 | 作用 |
|------|------|
| `label` / `object_id` | 给谁听、多目标时选哪个 |
| `image_uv` | 调试用；可做视觉伺服 |
| `bearing` + `range_m` | 单目常见：**方位角 + 估计距离**（精度有限，需标定或学习辅助） |
| `pose_vehicle` | **车体坐标系**下物体相对位置（最方便底盘直接走） |
| `pose_map` | 若有 SLAM/地图，可直接给全局规划 |
| `quality` | `mono_estimate` / `stereo` / `apriltag` 等，便于 agent 决策是否再靠近拍照 |

### 5.2 如何从「图像」到「能到达」

常见技术路径（按成本从低到高）：

1. **单目 + 固定地面假设 + 相机高度与俯仰角**：把检测框底边与地面交点近似成车体平面上的点，得到 \((x,y)\) 粗估计。  
2. **双目 / RGB-D**：深度可靠，`range_m` 与 `pose_vehicle` 精度明显提升。  
3. **AprilTag / ArUco**：若物体可贴码，定位到厘米级，适合 demo。  
4. **地图 + AMCL**：识别结果先变到 map，再由全局路径规划到目标栅格。

### 5.3 原子接口建议

- `vision_detect(image_id, target_hint)` → 上表结构列表（可多目标）。
- 可选 `vision_track(object_id)` → 连续帧更新 `pose_vehicle`（跟车移动时）。

## 6. 坐标系约定（设计阶段就要定死）

| 坐标系 | 原点与轴向 | 谁维护 |
|--------|------------|--------|
| `map` | 地图原点，通常右手系 | SLAM / 定位模块 |
| `vehicle` | 车体中心或后轴中心，**x 前、y 左、θ 逆时针** 等（团队内统一一处文档） | 底盘里程 + 外参 |
| `camera` | 光心 | 标定文件 |
| `gripper` | 夹爪尖端 | 正运动学 |

所有识别结果最终应能转换到 **`vehicle` 或 `map`**，底盘规划只消费这两种之一。

---

## 7. 与 tiny-agent 的衔接（后续实现方向）

- **工具拆分**：`mobile_base`（底盘）、`arm`（臂）、`camera`（拍照）、`vision`（检测+定位），与当前 `car_control` 可并存或合并为 `robot_*` 系列。  
- **Skill**：用一篇 toolkit 内 skill（例如 `toolkits/cars/skills/<name>/SKILL.md`）写清安全顺序（先停再臂、识别后再靠近等）。  
- **仿真**：先在 Gazebo / Webots 或纯 Python 假硬件里返回假 `pose_vehicle`，再换真机。

## 8. 未决问题（下一轮可拍板）

1. 底盘选 **方案 A** 还是 **方案 B**？  
2. 摄像头固定车体还是腕载？是否上 RGB-D？  
3. 室内是否有 **先验地图**；没有则前几周用「里程 + 识别相对位姿」即可。  
4. 识别模型：只分类 / 要实例分割 / 是否要 6D 姿态（抓取强相关）。

定稿以上四点后，可以把 JSON 字段冻结成 v1 协议，再写具体驱动或 HTTP 网关。
